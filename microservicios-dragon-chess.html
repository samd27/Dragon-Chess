Propuesta de Microservicios para Dragon Chess

Este documento describe que microservicios se agregarian, que tecnologias se usarian y como se conectan, siguiendo el diagrama propuesto: Frontend -> API Gateway -> servicios de Auth/Perfil, Game Economy, Content/Inventory, Chess Engine y Email/Notificaciones.

1) Vision general de arquitectura

El cliente (web o movil) se conecta al Frontend (React + Inertia). El Frontend llama a un API Gateway. El API Gateway enruta cada solicitud al microservicio correcto.

Cada microservicio tiene su propia base de datos para reducir acoplamiento. Redis se usa para cache, locks y estado temporal. RabbitMQ o Kafka se usa como bus de eventos para comunicacion asincrona. Todo se despliega en Railway como servicios separados por dominio.

2) Microservicios que se agregarian

2.1 Game Economy

Que es: Servicio de progreso y economia del jugador.

Que hace: Calcula EXP, Ki, Senzu, nivel, desbloqueos y compras.

Tecnologias a usar (Railway): Laravel API (PHP 8.2), PostgreSQL de Railway, Redis de Railway, REST interno y eventos con RabbitMQ (si se habilita asincronia).

Responsabilidades:

  • Procesar resultado de partida (win/loss/draw).
  • Aplicar recompensas y penalizaciones.
  • Subir nivel y desbloquear recompensas de battle pass.
  • Validar compras de personajes con recursos del jugador.

2.2 Content/Inventory

Que es: Servicio de catalogo de contenido y gestion de inventario.

Que hace: Administra personajes, precios, temporadas y disponibilidad.

Tecnologias a usar (Railway): Laravel API o Node.js (NestJS), PostgreSQL de Railway, Redis de Railway para cache y almacenamiento de assets en Cloudinary o S3 compatible.

Responsabilidades:

  • Publicar catalogo oficial de personajes.
  • Definir items de tienda y reglas de battle pass.
  • Versionar contenido por temporada.
  • Servir metadatos y rutas de assets.

2.3 Chess Engine (Go + Stockfish)

Que es: Servicio de motor de ajedrez para decisiones de CPU.

Que hace: Recibe estado del tablero (FEN) y devuelve bestmove.

Tecnologias a usar (Railway): Go 1.22+, binario Stockfish dentro del mismo servicio, API REST interna (o gRPC), Redis para colas cortas y control de concurrencia por workers.

Responsabilidades:

  • Ejecutar Stockfish como proceso controlado por Go.
  • Aplicar niveles de dificultad (depth/movetime/skill).
  • Responder movimiento de CPU de forma consistente.
  • Permitir escalado horizontal por workers.

2.4 Email/Notificaciones

Que es: Servicio de comunicacion con el jugador.

Que hace: Envia emails y notificaciones in-app/push.

Tecnologias a usar (Railway): Laravel Notifications o Node.js, Redis de Railway para colas, proveedor externo de email (Resend, SendGrid o Mailgun) y WebSockets para notificaciones en tiempo real.

Responsabilidades:

  • Enviar avisos de progreso, recompensas y eventos.
  • Procesar colas de envio asincrono.
  • Manejar reintentos y trazabilidad de envios.

2.5 Auth y Perfil (servicio existente, centralizado al inicio)

Que es: Servicio de identidad y perfil de usuario.

Que hace: Login, registro, sesion, perfil y datos base del usuario.

Tecnologias a usar (Railway): Laravel actual con Sanctum, PostgreSQL de Railway y almacenamiento de avatar en proveedor externo (Cloudinary/S3).

En la primera fase puede mantenerse en Laravel actual y luego evolucionar si se requiere separar identidad.

3) Tecnologias propuestas y razon de uso

3.1 Plataforma y despliegue

  • Docker + Docker Compose: Entornos reproducibles para desarrollo local.
  • Railway: Despliegue rapido por servicio, variables de entorno por proyecto y databases administradas.
  • API Gateway (BFF en Laravel o servicio dedicado): Entrada unica, ruteo, seguridad y rate limit.

3.2 Datos y cache

  • PostgreSQL por servicio (Railway): Aislamiento de datos por dominio.
  • Redis (Railway): Cache, locks distribuidos, colas rapidas y estado temporal.
  • Object Storage (S3/compatible): Assets y contenido estatico de personajes.

3.3 Comunicacion entre servicios

  • REST: Operaciones CRUD y consultas estandar.
  • gRPC (interno opcional): Baja latencia para servicios criticos (ejemplo Chess Engine).
  • RabbitMQ: Eventos asincronos entre servicios en fase inicial (Kafka solo si el volumen crece mucho).

3.4 Observabilidad y calidad operativa

  • OpenTelemetry: Trazas distribuidas.
  • Prometheus + Grafana (opcional por fase): Metricas y dashboards.
  • Sentry: Monitoreo de errores.
  • CI/CD (GitHub Actions): Build, tests y despliegue automatico.

4) Flujo operativo alineado al diagrama

  1. El jugador inicia una partida desde Frontend.
  2. El Frontend consulta al API Gateway.
  3. Para CPU, el Gateway llama al Chess Engine.
  4. Chess Engine pide jugada a Stockfish y devuelve bestmove.
  5. Al finalizar partida, Game Economy calcula recompensas.
  6. Game Economy publica eventos (ejemplo: RewardGranted).
  7. Email/Notificaciones consume eventos y envia mensajes al usuario.
  8. Content/Inventory entrega reglas de catalogo, precios y battle pass.

5) Beneficios esperados

  • Escalado independiente por dominio (engine, economy, notificaciones).
  • Menor acoplamiento entre logica de juego y contenido.
  • Mejor control de rendimiento y observabilidad.
  • Base preparada para temporadas, matchmaking y crecimiento futuro.

6) Resumen final

La arquitectura recomendada usa API Gateway + microservicios de Game Economy, Content/Inventory, Chess Engine (Go + Stockfish) y Email/Notificaciones, con Redis, PostgreSQL y bus de eventos. El despliegue se realiza en Railway con servicios separados por dominio. Esto permite evolucionar Dragon Chess de forma escalable sin perder la base actual de Laravel + React.
